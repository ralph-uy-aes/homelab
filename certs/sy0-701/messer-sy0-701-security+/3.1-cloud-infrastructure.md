# 3.1 Cloud Infrastructure

#### Cloud responsibility matrix
- IaaS, PaaS, SaaS, etc
    - Who's responsible for security?
- Security should be well documented
    - Most cloud providers have a matrix of responsibilities
    - Everyone knows up front
- These responsibilities can vary
    - Different cloud providers
    - Contractual agreements

#### Hybrid Considerations
- Hybrid cloud
    - More than one cloud being used
    - Adds additional complexity
- Network protection mismatches
    - Authentication across platforms
    - Firewall configurations
    - Server settings
- Different security monitoring
    - Diverse logs
- Data leakage
    - More openings for attack with more providers

#### Third-party vendors in the cloud
- Cloud providers and third-party vendors
    - Infrastructure technologies
    - Cloud-based appliances
- Ongoing vendor risk assessments
    - Part of an overall vendor risk management policy
    - Manage and maintain security for third party products
- Include third-party impact for IR
    - Everyone is a part of the process
- Constant monitoring
    - Watch for changes and unusual activities

#### Infrastructure as code
- Describe an infrastructure
    - Define servers, network, and applications as code
- Modify the infrastructure and create versions
    - Same way you version application code
- Use description to build other application instances
- An important concept for cloud computing
    - Build a perfect version every time

#### Serverless architecture
- Function as a Service (FaaS)
    - Accessing individual functions instead of entire application
    - Remove the OS from the equation
- Developer still creates server-side logic
    - Runs in a stateless compute container
- May be event triggered and ephemeral
    - May only run for one event
- Managed by a third party
    - All OS security concerns are third-party

#### Microservices and APIs
- Traditionally, we used Monolithtic applications
    - One big app does everything
- Makes all decisions
    - User interface
    - Business logic
    - Data I/O
- Code challenges
    - Large codebase
    - Change control challenges
- APIs help bypass this
    - Application Programming Interfaces
    - Instead of one big .exe, we can break down individual services per application
    - One instance in the cloud per microservice
    - APIs are the glue for microservices
- Scalable 
    - Scale just the microservices you need
- Resilient
    - Outages are contained
- Security and compliance
    - Containment is built-in
