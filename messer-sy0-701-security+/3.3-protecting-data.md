# 3.3 Protecting Data

#### Geographic Restrictions
- Network Location
    - Identify based on IP subnet
    - Can be difficult with mobile devices
- Geolocation - determine a user's location
    - GPS in mobile devices, very accurate
    - 802.11 wireless, less accurate
    - IP address, not very accurate
- Geofencing
    - Automatically allow or restrict access when user is not in a facility/building
    - Don't allow app to run if they're not in-office

#### Protecting Data
- Primary goal
    - An organization goes out of business if it loses its data
- Data is everywhere
    - On a storage drive, on network, or in a CPU
- Protecting the data
    - Encryption, security policies
- Data permissions
    - Not everyone has the same access

#### Encryption
- Turning plaintext into ciphertext (unreadable data)
- Two way street, meaning can be converted back and forth mathematically
- Ciphertext cannot be similar or identical to plaintext

#### Hashing
- Turning plaintext into a mathematically related string
    - Comparable to a human fingerprint
- One way, cannot use a hash to form the original plaintext via the same hashing algorithm
- Verify downloaded document is the same as the original
    - Integrity
- Can be used as a digital signature
    - Authentication, non-repudiation, integrity
- Ideally no collisions
    - Different messages do not have the same hash

#### Obfuscation
- Turning something easy to understand by humans into nonsense
- A developer could create readable code and give you the same code ran through an obfuscator
    - Both sets of code perform exactly the same
- Helps prevent search for security holes
    - More difficult to figure out what's happening
- Some software can deobfuscate, so not airtight

#### Masking
- More pragmatic type of obfuscation
    - Hide some original data, optionally show some
- Protects PII and other sensitive data
- Think of asterisks blocking everything but the last 4 digits of a credit card number
- Only usually hidden in plain view
    - Data may still be intact in storage
    - Control view based on permissions
- Many different techniques
    - Substituting, shuffling, encrypting, masking out, etc.

#### Tokenization
- Replace sensitive data with a non-sensitive placeholder
    - Tokenizing an SSN "266-12-1112" into "111-22-3333" and transmitting that over the network
    - Trick is that this token is one time use, and attackers capturing it cannot replay it to the server
- Common with credit cards in payment terminals
- Interestingly, no encryption or hashing
    - No encryption/decryption overhead
- Process:
    - Initially, user has a credit card registered on their phone
    - Credit card number is registered to a Remote Token Service Server
    - RTTS sends back a token to the phone
    - Phone is used at checkout and the token is used
    - Token is sent to the Merchant Payment Processing Server
    - MPPS relays the token to RTTS and validates the token


#### Segmentation
- Basically, not putting all your eggs in one basket
    - Don't put all your data in one database
- If an attacker gets into that single data source, all your data is gone
    - Instead, store data in different, separate locations
- Sensitive data should have stronger security
    - Most sensitive data should be most secure

#### Permission Restrictions
- Controlling access per account
    - Beyond "login credentials"
- Authentication process must take multiple distinct policies into account
- After logging in, users are further restricted by what they can access
    - Typically why whaling attacks occur, executives typically have less restrictions
    - Although, lower privilege accounts can still climb restrictions ladder
