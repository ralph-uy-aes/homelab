# 3.4 Resiliency

#### High Availability
- Buy multiple systems to keep services available (redundancy)
- Concept of HA (high availability)
- Often means many different components working together
- This comes with higher costs, redundancy can get expensive
    - Contingencies means more devices means more money spent
    - Upgrades mean more money spent as well

#### Server Clustering
- Multiple servers configured to work together as one big server
    - Shared storage
- Customers only see a cluster
- Highly scalable
    - Increases capacity and availability
- Usually configured in the OS
    - Typically, all servers use the same OS as well

#### Load Balancing
- Distributes server load betweens multiple servers
- Servers are isolated, they think they're standalone
- Highly scalable as well, servers can be added or removed
- If a server fails, all its load will be rerouted by the load balancer into a different one

#### Site Resiliency
- Prepare a recovery site 
    - If a site fails, move to the recovery until the original site is fixed

#### Hot Site
- A type of recovery site
- It is an exact duplicate or replica of the original site
    - "Buy two of everything"
- Stocked with hardware that is constantly updated
    - Applications and software are updated as well
- Flip a switch upon site failure and the Hot Site takes over

#### Cold Site
- Better for organizations who don't expect a lot of catastrophic site failure
- No hardware, typically an empty building
- No data, must be brought separately
- No staff

#### Warm Site
- The in-between
- Has some bits and pieces, but is incomplete
- Some things from the original site must be brought for operations

#### Geographic Dispersion
- "Don't put all your data centers in one city"
- Concept of keeping a recovery site away from the original site geographically
- Typhoons, hurricanes, etc can disrupt operations in an entire city
- Can be a logistical challenge
    - Transporting equipment
    - Getting employees on-site
    - Recovering back post-failure

#### Platform Diversity
- "Don't put all your software in one OS"
- Every OS has vulnerabilities, better to use many in case one fails
- Limits exposure to a single vulnerability
- However, this also increases attack surfaces
    - Now you can be hit by Windows, MacOS, or Linux security attacks

#### Multi-cloud Systems
- "Don't put all your data in one cloud provider"
- Having more cloud providers allows you to avoid downtime in case one cloud provider is down
- Data is geographically dispersed and cloud-service dispersed
    - A security breach in one provider doesn't affect the others
- Likewise, increases likelihood of supply chain attack

#### Continuity of Operations Planning (COOP)
- If everything fails and no backup services are available, a COOP is used as a fallback
- There needs to be an alternative
    - Manual transactions
    - Paper receipts
    - Phone calls for transaction approvals
- These policies and procedures must be documented and tested before a problem does occur
