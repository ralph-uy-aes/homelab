# 3.4 Recovery Testing

#### Recovery Testing
- Test yourselves before an actual event
    - Scheduled update sessions (annual, semi-annual, etc)
- Use well-defined rules
    - Of course, don't disrupt production systems
- Limit the time or track it, see how fast you can recover
- Evaluate response and discuss improvements

#### Tabletop Exercise
- Disaster drills can be costly and time consuming
- Instead, do a tabletop exercise
    - Basically a simulation with other people
    - More of an analysis

#### Fail Over Test
- Failures will occur
- See if your redundancies work
    - Is the switch smooth?
    - Does it work?
    - Is it automatic?

#### Simulation
- Check if automated systems can detect simulated cyber attacks
- Simulated phishing
    - Send a simulated phishing email to all of your employees
    - Check if phishing filter blocks it first
    - Then, see which employees interact with it and retrain them

#### Parallel Processing
- Splitting a process through parallel CPUs
    - Single computer with multiple CPU cores or multiple physical CPUs
- Improves performance
- Improves recovery
    - Identify which process failed under what processor
    - Offload processes to other good processors
